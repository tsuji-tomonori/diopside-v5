ドキュメント更新

## 計画

3. 変更範囲に見合う検証を実行する。
4. 作業レポート、commit、PR、受け入れ条件コメント、セルフレビューまで完了する。

## ドキュメント保守方針

運用手順変更のため README を主成果物とし、CloudFormation resource 名と実際の queue 構成に合わせて記載する。実 AWS 操作は未実施として明記する。

## 受け入れ条件

- README に対象 queue / DLQ の一覧が記載されている。
- README に DLQ depth の確認手順が記載されている。
- README に DLQ message の原因調査手順が記載されている。
- README に安全な再投入手順が記載されている。
- README に破棄判断と破棄前後の記録手順が記載されている。
- README に再投入後の確認と再発防止観点が記載されている。
- 実 AWS 操作を行っていない場合、その制約が作業レポートと PR に明記されている。
- 変更範囲に見合う検証が成功する。

## 検証計画

- `git diff --check`
- README 差分の目視確認
- 必要に応じて `npm test`

## PRレビュー観点

- 実装済み CloudFormation resource 名と README の queue / DLQ 名が一致していること。
- 実施していない AWS 操作を実施済みと書いていないこと。
- 再投入や破棄が不可逆操作であることを明確にしていること。

## リスク

- AWS CLI の具体コマンドは deploy stack 名や環境名に依存するため、README では resource 名と確認観点を中心にする。
- 実 AWS DLQ 操作は行わないため、手順は静的確認に留まる。
