# JobEvent v0.4 shape 対応

## 背景

`.workspace/plan-20260530.txt` の v0.4 設計準拠対応では、DDB item schema の差分解消が残っている。`docs/design/dynamodb-schema-audit.md` では `JobEvent` が `EVENT#{time}#{uuid}` / `event_type` / `details` の現行形状で、v0.4 設計の `EVT#{seq}` / `event_name` / `state_after` と差分ありと記録されている。

## 目的

`JobEvent` の新規保存形状を v0.4 設計に寄せ、管理 API や worker が使う既存の `append_job_event` 呼び出しを壊さずに、状態導出と履歴取得が新旧形状を扱えるようにする。

## タスク種別

機能追加

## スコープ

- `apps/shared/src/diopside_core/repository.py` の `JobEvent` 保存・読み取り・状態導出。
- `tests/test_repository_schema_contract.py` の DDB schema contract。
- `README.md` と `docs/design/dynamodb-schema-audit.md` の JobEvent 形状記述。
- 作業レポート、PR コメント、task done 更新。

## スコープ外

- `Job` item 自体の GSI3 / read model 完全移行。
- DynamoDB 既存データの backfill / migration。
- Worker の物理分割や外部通知 delivery。

## 実施計画

1. 現行 `append_job_event` と `derive_job_state` の形状依存を確認する。
2. `JobEvent` 新規 item を `EVT#{seq}` / `event_name` / `state_after` / `occurred_at` / `payload` で保存する。
3. 既存 `event_type` / `details` 形状の読み取り互換を残す。
4. schema contract test と audit / README を更新する。
5. targeted test、docs consistency、diff check、必要に応じて全体 verify を実行する。

## ドキュメントメンテナンス方針

`README.md` の item schema 表と `docs/design/dynamodb-schema-audit.md` の `JobEvent` 行を更新する。設計書本体 `docs/design/diopside_basic_design_v0.4.md` は既に v0.4 形状を記載しているため、実装後に差分 audit 側を更新する。

## PR レビュー観点

- 新規 `JobEvent` が v0.4 設計の key/field に寄っていること。
- 既存 worker / API 呼び出しが `append_job_event(job_id, event_type, details)` のまま壊れないこと。
- 旧形状のイベント履歴がある場合でも状態導出が破綻しないこと。
- 未対応の backfill / migration を実施済みと誤記していないこと。

## リスク

- 既存 DynamoDB に旧形状 `EVENT#...` が残る場合、完全な物理キー移行には backfill が必要。
- `seq` は現行 repository query 結果から採番するため、同一 job へ高並列 append する場合は DynamoDB 条件付き採番などの追加設計が必要。
